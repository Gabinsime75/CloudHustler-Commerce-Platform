# 🚀 ACM Certificate Module

The **ACM Certificate Module** provisions an AWS Certificate Manager certificate using reusable Terraform code.

This module supports public ACM certificates, DNS validation, email validation, Subject Alternative Names, wildcard certificates, Route53 DNS validation records, and certificate validation waiting.

---

## 🎯 Purpose

This module creates and manages ACM certificates for secure HTTPS communication across the CloudHustler Commerce Platform.

It is designed to support:

- Application Load Balancers
- CloudFront Distributions
- API Gateway Custom Domains
- Internal and External HTTPS Endpoints
- Wildcard Certificates
- Multi-Domain Certificates

## 📦 Resources Created

This module can create the following resources:

- `aws_acm_certificate`
- `aws_route53_record`
- `aws_acm_certificate_validation`

Route53 records and ACM certificate validation are only created when DNS validation and Route53 record creation are enabled.

## 📥 Inputs

| Name | Description | Type | Default |
|------|-------------|------|---------|
| `domain_name` | Primary domain name for the ACM certificate. | `string` | Required |
| `subject_alternative_names` | Subject Alternative Names for the certificate. | `list(string)` | `[]` |
| `validation_method` | Certificate validation method. Valid values are `DNS` or `EMAIL`. | `string` | `DNS` |
| `create_route53_records` | Whether to create Route53 DNS validation records automatically. | `bool` | `true` |
| `hosted_zone_id` | Existing Route53 Hosted Zone ID used for DNS validation. | `string` | `null` |
| `hosted_zone_name` | Route53 Hosted Zone name used for lookup when hosted zone ID is not provided. | `string` | `null` |
| `allow_overwrite_validation_records` | Whether Route53 validation records may overwrite existing DNS records. | `bool` | `true` |
| `validation_record_ttl` | TTL for Route53 DNS validation records. | `number` | `60` |
| `key_algorithm` | Key algorithm for the ACM certificate request. | `string` | `RSA_2048` |
| `certificate_transparency_logging_preference` | Certificate Transparency logging preference. Valid values are `ENABLED` or `DISABLED`. | `string` | `ENABLED` |
| `tags` | Tags applied to supported resources. | `map(string)` | `{}` |

## 💻 Example Usage

### DNS Validation with Hosted Zone ID

```hcl
module "acm" {
  source = "../modules/acm"

  domain_name = "cloudhustler.com"

  subject_alternative_names = [
    "www.cloudhustler.com",
    "api.cloudhustler.com"
  ]

  validation_method      = "DNS"
  create_route53_records = true
  hosted_zone_id         = "Z123456789ABCDEFG"

  tags = {
    Project     = "cloudhustler-commerce-platform"
    Environment = "dev"
    ManagedBy   = "Terraform"
  }
}
```

### External DNS Provider

Use this pattern when DNS is managed outside AWS, such as Cloudflare, GoDaddy, or Namecheap.

```hcl
module "acm" {
  source = "../modules/acm"

  domain_name = "cloudhustler.com"

  subject_alternative_names = [
    "www.cloudhustler.com"
  ]

  validation_method      = "DNS"
  create_route53_records = false

  tags = {
    Project     = "cloudhustler-commerce-platform"
    Environment = "dev"
    ManagedBy   = "Terraform"
  }
}
```

After applying, use the `domain_validation_options` output to manually create the required DNS validation records in your external DNS provider.

## 🔒 Security Considerations

- DNS validation is recommended for automated certificate validation.
- Email validation requires manual approval and should be avoided for fully automated infrastructure pipelines.
- Use wildcard certificates carefully and only when they are operationally justified.
- Keep Certificate Transparency logging enabled unless there is a specific security or privacy reason to disable it.
- Use least-privilege IAM permissions for Route53 record creation.
- Avoid sharing private hosted zone IDs across unrelated environments.
- Use separate certificates for different environments when isolation is required.

---

## 💡 Design Decisions

- The module requests public ACM certificates only.
- Imported certificates and ACM Private CA certificates are intentionally excluded.
- DNS validation records are optional to support external DNS providers.
- Route53 hosted zone lookup is optional.
- `create_before_destroy` is hardcoded in the certificate lifecycle for safe replacement.
- Route53 validation records are only created when DNS validation and Route53 integration are enabled.
- The module exposes validation options so DNS records can be created outside AWS if needed.

## 🧪 Testing

Run the following commands from the module directory:

```bash
terraform fmt
terraform validate
```

For integration testing, consume this module from a root module and run:

```bash
terraform init
terraform plan
```

If using Route53 DNS validation, confirm that:

- The hosted zone exists.
- The hosted zone ID or hosted zone name is correct.
- Terraform can create Route53 records.
- The certificate reaches `ISSUED` status.
